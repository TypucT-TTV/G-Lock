# Типы сессий — подробный разбор

Общий принцип для всех пунктов ниже (кроме "Кикнуть неизвестных" и "Кикнуть
по IP", это разовые действия): нажатие кнопки → диалог подтверждения с
описанием режима → при согласии создаётся объект фильтра, регистрируется в
`Context` (`add_filter`) и запускается (`start_latest_filter`), что
автоматически останавливает предыдущий активный фильтр, если он был.

## 1. Solo-сессия (`SoloSession`)

**Что происходит технически:** `is_packet_allowed` пропускает **только**
пакеты с размером payload из `HEARTBEAT_SIZES` (`{12, 18, 63}`). Вообще всё
остальное входящее на порт 6672 — отбрасывается.

**Практический эффект:** никто не может зайти к вам в сессию. Если в момент
включения в сессии уже были другие игроки — они потеряют соединение с вами
(не с игрой в целом, а конкретно с вашим клиентом), потому что весь трафик от
них, кроме heartbeat, будет дропаться. Критичные сервисы Rockstar/Social Club
продолжат работать (их трафик обычно не проходит через порт 6672 таким
образом, либо укладывается в heartbeat-размеры).

**Когда использовать:** хотите играть полностью в одиночестве, максимально
строгий режим защиты.

## 2. Сессия по вайтлисту (`WhitelistSession`)

**Что происходит технически:** список IP из вашего Whitelist (см.
`04_Списки_вайтлист_блэклист.md`) сначала валидируется (`IPValidator.
validate_get` — проверка формата IPv4; невалидные записи пропускаются с
предупреждением в лог, но не останавливают запуск). Затем `is_packet_allowed`
пропускает пакет, если:
- IP отправителя есть в списке, **или**
- размер — heartbeat, **или**
- сработала дополнительная байтовая эвристика (XOR-проверка конкретных
  смещений payload — резервный механизм на случай туннелирования через
  IP, которого нет в списке напрямую, см. `00_Обзор_архитектуры.md`).

**Практический эффект:** заходить смогут только адреса из вашего Whitelist.
Если вы **хост** сессии — все, кого нет в списке, отвалятся. Если вы **не
хост**, и кто-то из участников сессии не в вашем Whitelist — отвалитесь
именно **вы** (у вас нет прав хоста, чтобы кикать других, поэтому рвётся
именно ваше соединение с ними).

**Когда использовать:** хотите играть с конкретными друзьями, чьи IP вы
заранее знаете и добавили в список.

## 3. Сессия по блэклисту (`BlacklistSession`)

**Что происходит технически:** более сложная логика с кэшированием:
- Если IP в `known_allowed` (изначально туда попадают ваш публичный и
  локальный IP) или размер пакета "известный" (`HEARTBEAT_SIZES ∪
  MATCHMAKING_SIZES`) — пропуск сразу.
- Если IP явно есть в вашем Blacklist — блок.
- Если IP **не** в блэклисте напрямую, но попадает в один из CIDR-диапазонов
  динамического блэклиста (список подсетей Rockstar/T2 + Microsoft Azure,
  подробнее — `06_Обработка_ошибок.md`, раздел про сетевые запросы при
  старте) — блок, и адрес **запоминается** в `self.ips`, чтобы при повторной
  встрече не пересчитывать CIDR-проверку заново.
- Если нигде не найден — пропуск, и IP запоминается в `known_allowed` (тот же
  смысл: не пересчитывать один и тот же адрес много раз за сессию).

**Практический эффект:** блокируются конкретные нежелательные адреса из вашего списка. Этот режим назначен **фоновым режимом по умолчанию**. Он запускается автоматически при старте программы, при остановке любого другого фильтра, а также при разблокировке сессии клавишей F9.

Чтобы избежать случайной блокировки матчмейкинг-релеев Rockstar (что ломало подключение вашим друзьям), фоновый черный список по умолчанию **исключает динамический блэклист Rockstar/Azure** (`Menu.dynamic_blacklist`) и фильтрует трафик только по вашему персональному блэклисту. Динамический блэклист по-прежнему можно активировать через явный выбор режима в меню.

## 4. Авто-вайтлист сессии

**Что происходит технически (сложный многошаговый сценарий):**
1. Подтверждение.
2. 60 секунд сбора IP через `IPCollector` (см. ниже "Технические детали
   сбора IP") — с прогресс-баром и возможностью отмены.
3. Среди собранных IP ищутся **потенциальные туннели**: адрес попадает в этот
   список, если он входит в CIDR-диапазоны динамического блэклиста
   (Rockstar/Azure) **и** ещё не находится в вашем постоянном Whitelist.
4. Если такие найдены — показывается предупреждение с чекбоксами: можно
   выбрать, каких из "подозрительных" всё-таки оставить в сессии (по
   умолчанию все сняты — то есть по умолчанию подозрительные адреса
   **исключаются** из итогового набора).
5. Из оставшихся (обычных + явно оставленных подозрительных) собранных IP
   создаётся `WhitelistSession` и запускается.

**Практический эффект:** все, кто **сейчас** в сессии, временно получают
доступ (без ручного добавления в постоянный Whitelist) — удобно, когда вы уже
играете с кем-то и просто хотите закрыть сессию от новых людей, не вводя
IP-адреса вручную. Собранные адреса **не сохраняются** — при следующем
включении сбор произойдёт заново.

**Риск:** если разрешить "подозрительный" (потенциально туннелированный)
адрес остаться — защита сессии может быть скомпрометирована, так как через
такой туннель потенциально может пройти кто угодно.

## 5. Закрытая сессия (Lock) (`LockedSession`)

**Что происходит технически:** `is_packet_allowed` блокирует **только**
пакеты размером из `MATCHMAKING_SIZES` (`{191, 207, 223, 239}`) — то есть
новые попытки зайти в сессию. Всё остальное (включая трафик от уже
подключённых игроков) пропускается без изменений.

**Практический эффект:** новых игроков в сессию не пускает, но никого не
кикает. Единственный режим, специально предназначенный, чтобы отсечь вход
именно через "туннель" Rockstar-серверов, при этом не трогая тех, кто уже
внутри (даже если их трафик тоже шёл через такой туннель). Если кто-то из
уже подключённых **выйдет** — обратно зайти не сможет, пока лок активен.

**Это единственный режим, который также переключается клавишей F9** —
подробности в `03_Горячая_клавиша_F9.md`. Кнопка в интерфейсе и F9 управляют
одним и тем же состоянием: индикатор на кнопке "Закрытая сессия" загорается и
гаснет одинаково, независимо от того, чем именно был вызван переключение.

## 6. Кикнуть неизвестных

**Что происходит технически:** это не постоянный режим, а разовое временное
действие:
1. Подтверждение.
2. Запускается `WhitelistSession` (по текущему Whitelist) — все, кого нет в
   списке, начинают отваливаться.
3. Прогресс-бар на 10 секунд (`Menu._wait_with_progress`), с возможностью
   отмены (отмена просто сокращает ожидание, не отменяет сам кик).
4. По истечении — вызывается метод `Menu.stop_session()`, который снимает временный фильтр и восстанавливает фоновый блэклист по умолчанию.

**Официально не рекомендуется** — как и с блэклистом, клиенты потенциально
могут пытаться маршрутизировать трафик неизвестного игрока через IP, который
как раз есть в вашем Whitelist, обходя смысл кика.

## 7. Пустая сессия (стать хостом) (`launch_new_session`)

**Что происходит технически:** технически идентично Solo-сессии — создаётся
точно такой же `SoloSession`. Единственная разница — другая кнопка/текст
подтверждения, рассчитанные на другой сценарий использования: "мне нужно
гарантированно оказаться единственным в сессии, чтобы стать её хостом", а не
"хочу играть в одиночестве как самоцель". Индикаторы кнопок "Solo-сессия" и
"Пустая сессия" поэтому загораются вместе — это один и тот же фильтр.

## 8. Кикнуть по IP

**Что происходит технически:**
1. Подтверждение.
2. 60 секунд сбора IP через `IPCollector` (см. ниже), с прогресс-баром и
   отменой.
3. Показывается список собранных IP с чекбоксами — "Select IPs to kick" /
   "Выберите IP для кика". Отмеченные адреса **удаляются** из итогового
   набора допустимых.
4. Из оставшегося набора создаётся `WhitelistSession` — то есть отмеченные
   для кика адреса перестают быть в списке разрешённых и начинают отваливаться.
5. 10 секунд ожидания с прогресс-баром, затем автоматическое снятие фильтра с вызовом `Menu.stop_session()` для восстановления фонового блэклиста по умолчанию.

**Практический эффект:** позволяет выбрать конкретных игроков из текущей
сессии и временно выкинуть именно их, оставив остальных нетронутыми.

## Технические детали сбора IP (`IPCollector`)

Используется в пп. 4 и 8. Работает в режиме `SNIFF` (наблюдение без
блокировки — параллельно с любым другим активным фильтром или без него) в
течение заданного времени (по умолчанию везде — 60 секунд, можно прервать
досрочно кнопкой "Отмена", что просто укорачивает время сбора и продолжает
работу с уже собранным на тот момент):

- Считаются только **входящие** пакеты, размер которых **не** входит в
  `HEARTBEAT_SIZES ∪ MATCHMAKING_SIZES` — то есть явно "обычный игровой
  трафик", а не служебные пакеты или единичные попытки подключения.
- Для каждого такого IP ведётся счётчик пакетов.
- В финальный список попадают только адреса, от которых пришло **не менее
  15 пакетов** (`packet_count_min_threshold=15`) за время сбора — это
  отсекает случайные одиночные "пинги" (например, от служебных адресов R*),
  которые не являются реальными активными участниками сессии.
