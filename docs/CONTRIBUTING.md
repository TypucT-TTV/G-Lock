# Руководство по контрибуции в G-Lock v2

Мы рады вашему участию в улучшении G-Lock! Чтобы процесс внесения изменений прошел максимально гладко и быстро, пожалуйста, ознакомьтесь со стандартами разработки, принятыми в этом репозитории.

---

## 1. Процесс внесения изменений

1. **Создайте форк репозитория** и склонируйте его на локальный компьютер.
2. **Создайте тематическую ветку** от ветки `main`:
   ```bash
   git checkout -b feature/my-new-feature
   ```
3. **Внесите изменения в код** и напишите тесты (если применимо).
4. **Убедитесь в прохождении всех локальных проверок** (см. раздел "Стандарты кода").
5. **Закоммитьте изменения** и отправьте ветку в ваш форк:
   ```bash
   git push origin feature/my-new-feature
   ```
6. **Создайте Pull Request (PR)** в ветку `main` оригинального репозитория.

---

## 2. Стандарты кода

Качество и стиль кодовой базы проекта поддерживаются автоматически при помощи встроенных компиляторов и статических анализаторов. Перед созданием Pull Request обязательно запустите следующие проверки:

### 1. Форматирование и проверка Rust-кода (бэкенд)
Весь Rust-код должен быть отформатирован в соответствии со стандартами:
```bash
cargo fmt --all -- --check
```
И протестирован с помощью Clippy на отсутствие предупреждений и ошибок:
```bash
cargo clippy --all-targets -- -D warnings
```

### 2. Сборка фронтенда
Убедитесь, что Svelte 5 фронтенд собирается без ошибок и предупреждений линтера:
```bash
npm run build
```

---

## 3. Коммиты

- Пишите понятные и лаконичные сообщения к коммитам.
- Рекомендуется использовать формат коммитов Conventional Commits:
  - `feat: добавить новый режим фильтрации`
  - `fix: исправить баг масштабирования UI`
  - `docs: обновить руководство пользователя`

---

## 4. Лицензия

Создавая Pull Request, вы соглашаетесь с тем, что ваш код будет лицензирован по той же лицензии, что и весь проект G-Lock.
